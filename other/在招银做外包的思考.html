              # 在招银做外包的思考

              # 提测演示制度的弊端

              1.开发人员不可能完全把握所有细节

              2.提测演示时若疏忽了部分内容将导致提测失败

              3.不能及时提测的故事将需要开发自测

              4.提测率做为考核一个团队绩效的指标

              5.为了保证提测成功，开发人员面临在开发阶段就要实现所有细节的压力

              6.导致开发人员在需求实现阶段不能专注于主体功能的实现上，增加了开发成本，降低了开发效率

              7.需求实现阶段的延长也同时延迟了测试介入测试的时间点

              8.一个故事不能提测则不能开始另一故事的开发，影响了其他故事的开发

              # 改进措施

              1.移除提测演示制度，开发人员不可能完全不自测就进行提测

              2.或提测演示应以主体功能实现为主，细节问题不需要完全演示

              # 缺陷数量作为考核机制导致开发与测试对于缺陷的争论

              1.以缺陷密度作为考核一个开发团队的标准

              2.以缺陷提出数量作为考核测试人员的标准

              3.是不可取的

              4.为了减少缺陷，增大了开发人员在需求实现阶段的工作压力

              5.测试若以提出缺陷为目标，则不可避免的会减少对核心功能的测试

              6.在减少缺陷的道路上，开发与测试必不可少的会产生一些争论，这会时常打断双方的工作思路，降低了工作效率

              7.本质上应当以核心功能实现为主、细节部分完善为辅来开发与测试，不应本末倒置

              # 产品经理职能缺失的问题

              1.不能将需求准确的转化为业务逻辑，由此衍生出一系列问题

              2.需求实现的业务逻辑变的更复杂了

              3.需求实现的业务逻辑不稳定，经常变动

              4.需求实现的业务逻辑难以让人理解

              5.需求不断增加而导致页面臃肿

              6.需求实现的业务逻辑的合理性无人把控

              7.需求实现的业务逻辑的细节缺失

              # 基层领导不做开发只负责管理的思考

              1.降低了基层领导的开发水平

              2.基层领导的职位性质发生了变化，不再与基层成员同一阵线

              3.基层领导脱离了开发就不能够切身体会开发中遇到的问题

              4.基层领导由对内负责将逐步转变为对上负责

              5.基层领导将逐步散失领导力

              6.作为个人，基层领导不参与开发将降低自己的求职竞争力

              # 业务逻辑复杂度

              1.使用一个方案来统计业务逻辑的复杂度

              2.凡是能用一句简短的话归纳的则统计为一复杂度，比如点击登录按钮后登录用户、展示商品的信息

              3.业务逻辑中涉及到逻辑判断的，每条分支语句为一复杂度，比如当用户选择优惠券时总价中减去优惠的价格

              4.这样的话，就可以粗略地估计一下应用的业务逻辑复杂情况

              5.若某个页面或某块内容的业务逻辑过于复杂，就应当从业务逻辑上拆分为更小的单元

              # 关于外包制度

              1.个人感觉招银高层领导与基层领导对于外包员工的看法是分裂的

              2.高层领导眼中的外包员工的工作内容是不涉及核心内容的

              3.然而实际上，外包员工开始承担更多的职责，尤其在20年下半年到21年上半年

              4.由于本部员工离职，留下来的外包员工不得不承担着更多的责任

              5.显然在基层领导眼中，外包员工已经是组内不可分割的重要组成

              6.所以我认为外包制度不适合需要长期稳定但福利待遇不够高的团队

              7.首先在人才市场上很难招到优秀的外包员工

              8.这导致团队整体技术水平的降低，这可能导致本部原本优秀的员工的逃离

              9.周而复始，从而严重降低了了团队的技术水平

              10.况且外包员工在承担更多责任却福利待遇不及本部员工时，难免生出异心，使团队的凝聚力下降

              11.如果一个项目需要长期稳定的发展，那么采用外包制度是不可取的

              12.我也不希望以后我入职的公司采用外包制度，我需要一个强有力的技术团队

              # 项目何时开始走下坡路

              一个项目从什么时候开始走下坡路呢？

              从一个开发者的角度来考虑，那应该就是从开发人员无法掌握所负责应用的大部分业务逻辑时开始的

              为什么这么说呢？

              如果开发人员不能掌握他所负责的应用的大部分业务逻辑时，那么主要说明的就是人员短缺了，人员短缺明显就是一个项目走下坡路的标志

              如果一个项目重要的话，那应当多投入人力，如果明知项目重要却不愿意投入人力，导致无力去维护项目，那么项目会越来越难以开发，开发质量会越来越差，直至最后只能重构
